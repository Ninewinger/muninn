# Muninn Phase 11: Learning Pipeline Implementation Plan

## Overview
Implement the learning pipeline for Muninn to enable the system to learn new information from conversations. This transforms Muninn from a read-only semantic memory system to a dynamic learning system that can create new facets, update existing ones, and discover new peers automatically.

## Current State Analysis
- **Existing Architecture**: Router v2 with peer/facet activation system working
- **Database**: SQLite with vec0 for embeddings, full schema v2 implemented
- **Embeddings**: Qwen3-8B backend with singleton caching (4.6GB VRAM usage)
- **Dreaming**: Event processing system that creates memories from conversations
- **Integration**: Direct import-based (no HTTP servers)

## Goals
1. Enable automatic learning of new facts from conversation
2. Support three learning actions: create facet, update facet, create peer
3. Maintain data quality through similarity thresholds
4. Integrate seamlessly with existing dreaming consolidation
5. Provide testing infrastructure with mocked embeddings

## Implementation Strategy

### Core Components

#### 1. Learning Module (`muninn/learning.py`)
**Purpose**: Main learning logic for processing new facts
**Key Functions**:
- `learn_fact()`: Core learning function with similarity-based decision making
- `learn_batch()`: Batch processing for multiple facts
- `forget_facet()`: Manual facet removal
- `merge_facets()`: Consolidate similar facets

**Decision Logic**:
1. Embed new fact using existing embeddings_v2.py
2. Compare against all existing facet embeddings (cosine similarity)
3. If similarity > 0.85 with existing facet: update/merge that facet
4. If similarity > 0.15 with any peer's best facet: add new facet to that peer
5. If no relevant peer found: create new peer with single facet

#### 2. Enhanced Dreaming (`muninn/dreaming.py`)
**Modifications**:
- Add facet merging during consolidation
- Improve peer representation updates
- Add learning-triggered re-embedding
- Better cleanup of orphaned facets

#### 3. Testing Infrastructure (`tests/test_learning.py`)
**Requirements**:
- In-memory SQLite database for tests
- Mocked embeddings (no Qwen3 loading in tests)
- Test cases for all learning scenarios
- Integration tests with dreaming process

### Integration Points

#### With Nanobot
- Direct import: `from muninn.learning import learn_fact`
- Usage: `learn_fact("Diego started learning Rust", db_path="muninn.db")`
- No HTTP endpoints or servers

#### With Existing Router
- Reuse `router_v2.route()` for peer relevance scoring
- Leverage existing facet embedding comparison logic
- Maintain compatibility with current activation system

#### With Dreaming
- Learning can trigger dreaming consolidation
- Dreaming can trigger facet merging operations
- Shared embedding backend ensures consistency

### Technical Constraints

#### Hard Constraints
- **NO HTTP servers**: Integration via direct imports only
- **NO new embedding models**: Use existing Qwen3-8B singleton
- **NO schema changes**: Work within existing v2 schema
- **VRAM limits**: 16GB total, Qwen3 uses ~4.6GB, reranker ~560MB

#### Soft Constraints
- Maintain existing router_v2.py interface
- Follow current code patterns and error handling
- Use existing DB connection patterns
